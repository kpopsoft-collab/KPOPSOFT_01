# 어드민 기획서

작성: 2026-07-07 · 기준 문서: [기획서.md](기획서.md) · [디자인.md](디자인.md) · [스펙.md](스펙.md)

랜딩페이지를 운영하기 위한 내부 관리자(admin) 영역 기획. 랜딩페이지와 **같은 Next.js 앱, 같은 Supabase 프로젝트** 안에 `/admin` 경로로 둔다(별도 서비스 아님).

---

## 1. 목적 / 문제

지금 랜딩페이지는 두 가지 한계가 있다.

1. **문의가 남지 않는다.** Final CTA 폼이 `mailto:`로 메일 앱만 열어서, 문의가 DB에 쌓이지 않고 응대 상태도 추적 못 한다.
2. **콘텐츠 수정에 개발자가 필요하다.** work·insights·testimonials·experts·stats 등이 `src/lib/site.ts` 정적 데이터라, 후기 하나 바꾸려도 코드 배포가 필요하다.

어드민은 이 둘을 해결한다: **① 문의 접수·응대 관리**, **② 마케팅 콘텐츠 CMS**. 그 전제로 **③ 관리자 인증**이 필요하다.

---

## 2. 범위 (단계별)

한 번에 다 만들지 않고 3단계로 나눈다. 각 단계는 독립적으로 배포 가능.

| 단계 | 내용 | 산출물 |
|---|---|---|
| **P1 — 인증 + 문의 관리** | Supabase Auth 연결, 관리자 로그인, 문의 테이블 + 폼 연동, 문의 목록/상세/상태변경 | 실제로 문의가 쌓이고 응대 추적 가능 |
| **P2 — 콘텐츠 CMS** | site.ts의 정적 콘텐츠를 DB로 이관, 어드민에서 CRUD, 공개 페이지가 DB에서 렌더. **강사진 등록/삭제 + 강사 이미지 업로드(Storage) 포함**, **문의 폼 유형/세부 유형 관리 포함** | 비개발자가 콘텐츠 편집 |
| **P3 — 운영 편의** | 대시보드 요약, 관리자 초대 UI, 정렬 드래그, 활동 로그, insights 상세(블로그) | 실운영 품질 |

> **P1이 최우선.** P2·P3는 P1 안정화 후 착수. 아래 데이터 모델·화면은 전 단계를 담되 구현 순서는 이 표를 따른다.

## 3. 정보구조 / 라우트

App Router 기준. `/admin` 하위는 전부 인증 필요(레이아웃 가드).

```
/admin/login              로그인 (인증 전 유일 접근 지점)
/admin                    대시보드 (요약)
/admin/inquiries          문의 목록 (필터·검색)
/admin/inquiries/[id]     문의 상세 (상태 변경·메모)
/admin/content/work       Selected Work CRUD          ┐
/admin/content/insights   Insights CRUD               │ P2
/admin/content/testimonials  Testimonials CRUD        │
/admin/content/experts    Experts CRUD                │
/admin/content/stats      Company Numbers 편집        ┘
/admin/content/inquiry-options  문의 유형·세부 유형 CRUD   P2
/admin/settings           사이트 기본정보(email 등)   P3
```

- `/admin` 레이아웃: 좌측 사이드바 내비 + 상단 바(로그아웃, 계정). 모바일은 Sheet 드로어(기존 `ui/sheet.tsx` 재사용).
- 공개 사이트 네비에는 `/admin` 노출 안 함. 직접 URL 접근만.

**공개 라우트 추가(P2·P3):** 인사이트 블로그 상세
```
/insights            인사이트 목록(현재 랜딩 섹션과 별개의 전체 목록 페이지, 선택)
/insights/[slug]     인사이트 상세 글(블로그 본문)   ← 결정 §9
```
- 랜딩페이지 Insights 섹션 카드 → `/insights/[slug]`로 링크.

## 4. 데이터 모델 (Supabase / PostgreSQL)

### 4.1 `inquiries` — 문의 (P1)

| 컬럼 | 타입 | 비고 |
|---|---|---|
| `id` | uuid PK | `gen_random_uuid()` |
| `type` | text | `프로젝트 문의` / `교육 문의` / `AI 솔루션 문의` — 옵션 label 스냅샷 |
| `subtype` | text | 세부 유형 label 스냅샷(예: `웹 프로젝트`). 폼이 type+subtype 둘 다 제출 |
| `sender` | text | 이름/회사 |
| `contact` | text | 연락처(이메일/전화) |
| `message` | text | 문의 내용 |
| `status` | text | `new` / `in_progress` / `done` (default `new`) |
| `memo` | text | 관리자 내부 메모 |
| `created_at` | timestamptz | default `now()` |
| `updated_at` | timestamptz | 트리거로 갱신 |

- 인덱스: `status`, `created_at desc`.

### 4.2 콘텐츠 테이블 (P2) — site.ts 구조를 그대로 반영

공통 컬럼: `id uuid`, `sort_order int`(정렬), `is_published bool`(노출 토글), `created_at`, `updated_at`.

- `work_items` — client, title, category, accent, **`image_path` text**(커버 이미지, Storage. 없으면 도형 폴백)
- `insights` — tag, title, date, **`slug` text UNIQUE**(상세 URL), **`body` text**(본문·마크다운), **`image_path` text**(커버 이미지) — 목록 카드 + 블로그 상세 겸용(결정 §9)
- `testimonials` — quote, author, program, result
- `experts` — name, role, quote, tags(text[]), accent, **`image_path` text**(강사 사진 Storage 경로. 없으면 이니셜/도형 폴백)
  - **현 코드 상태**: 강사 3명 실데이터가 이미 들어감([site.ts](../src/lib/site.ts) `experts`, `Expert` 타입). 이미지는 아직 **Storage가 아니라 `public/experts/` 정적 파일**이고 필드명도 `image`(public 경로). P2에서 이 정적 파일들을 Storage `experts` 버킷으로 이관하고 필드를 `image_path`로 전환. 폴백 렌더는 이미 구현됨.
  - 정리 필요: `public/experts/`에 이관 전 원본(김상혁.png/안영근.png/안영근02.png, 1.6~1.7MB)이 남아 있음 → Storage 이관 시 사용본만 남기고 정리.
- `stats` — value(int), suffix, label
- `programs`, `businesses`, `software_categories` 도 **DB화 확정**. 단 변경 빈도 낮으므로 **P2 후반**에 이관(그전까지 site.ts 유지). accent는 기존 `Accent` 타입 7색 CHECK.

**문의 폼 옵션 (`inquiry_types`, `inquiry_subtypes`) — 문의 폼의 유형/세부 유형 선택지 (P2).** 섹션 콘텐츠(programs·businesses)와 값이 겹쳐도 **독립 관리**(결정 §9-3). 하드코딩 금지.

- `inquiry_types` — id, label(text), sort_order(int), is_active(bool)
- `inquiry_subtypes` — id, type_id(FK → inquiry_types), label(text), **`placeholder` text**(문의 내용 입력칸 예시 문구 — 세부 유형별로 다름, 폼 UX 핵심), sort_order(int), is_active(bool)
- 접수된 문의(`inquiries.type` / `inquiries.subtype`)는 옵션 label을 **문자열로 스냅샷 저장**(옵션이 나중에 바뀌어도 과거 문의 내용 보존). FK 강제 아님.
- 현재 시드([site.ts](../src/lib/site.ts) `inquiryOptions`)는 subtype당 `label` + `placeholder`를 가짐 → 스키마가 이 구조와 일치해야 무손실 이관. 세부 유형 6개 초과 시 폼이 드롭다운으로 전환(현 코드 `asDropdown = subtypes.length > 5`) — 어드민에서 개수 조절 시 이 임계값 인지.
- 현재 코드: [src/lib/site.ts](../src/lib/site.ts)의 `inquiryOptions`가 시드이자 단일 소스. 공개 폼([final-cta.tsx](../src/components/sections/final-cta.tsx))은 이미 이 배열만 읽도록 분리돼 있어, P2에서 **데이터 출처만 DB로 교체**하면 된다.

### 4.3 인증 / 권한

- **인증은 관리자 전용**(결정 §9). 스펙의 "회원가입/로그인 포함"은 이 프로젝트에서 **관리자 로그인**으로 해석 — 방문자용 일반 회원 개념 없음. 공개 회원가입 화면 만들지 않음.
- Supabase `auth.users` + `admin_users(user_id uuid PK, email text)` 테이블 + `is_admin()` SQL 함수.
- 관리자 계정은 **수동 부트스트랩**: 첫 계정은 Supabase 콘솔/시드로 생성 후 `admin_users`에 등록. 이후 추가는 어드민 초대 UI(P3).
- `auth.users`에는 관리자만 존재하므로, "일반 유저 vs 관리자" 분기 불필요 → RLS는 `is_admin()` 단일 게이트로 단순.

## 5. 접근 제어 (RLS) — 핵심

DB는 항상 RLS ON. 클라이언트 anon 키가 노출되는 전제로 정책을 짠다.

- **`inquiries`**
  - INSERT: `anon` 허용 (공개 폼이 직접 저장). 단, 컬럼 화이트리스트 + 서버 액션 경유 권장(스팸·검증).
  - SELECT / UPDATE / DELETE: `is_admin()` 인 경우만.
- **콘텐츠 테이블**
  - SELECT: `is_published = true` 는 anon 허용(공개 렌더), 전체는 admin.
  - INSERT / UPDATE / DELETE: admin만.
- **`inquiry_types` / `inquiry_subtypes`**
  - SELECT: `is_active = true` anon 허용(공개 폼 렌더), 전체는 admin.
  - INSERT / UPDATE / DELETE: admin만.
- 관리자 화면의 쓰기 작업은 **Server Action / Route Handler**에서 처리하고, 필요 시 service-role 키는 **서버에서만** 사용(절대 클라이언트 번들에 넣지 않음).

## 6. 화면별 설계

### 로그인 `/admin/login`
- 이메일/비밀번호(Supabase Auth). 관리자만 로그인 성공 → 아니면 거부 메시지.
- 성공 시 `/admin`. 세션은 `@supabase/ssr` 쿠키 기반(App Router 미들웨어로 갱신).

### 대시보드 `/admin`
- 카드형 요약: 신규 문의 수, 오늘 문의, 상태별 카운트. (P3에서 최근 문의 리스트·차트)

### 문의 목록 `/admin/inquiries`
- 테이블: 유형 / 세부 유형 / 이름·회사 / 연락처 / 상태 배지 / 접수일.
- 필터: 상태(new/in_progress/done), 유형(+세부 유형). 검색: sender/message.
- 상태 배지 색은 브랜드 accent 재사용(new=blue, in_progress=yellow, done=mint).
- 행 클릭 → 상세.

### 문의 상세 `/admin/inquiries/[id]`
- 원문 전체(유형·세부 유형 포함) + 상태 드롭다운(즉시 저장) + 내부 메모 textarea(저장). 연락처 mailto 링크.

### 콘텐츠 CRUD (P2)
- 각 리스트: 정렬(드래그 or sort_order 입력), 노출 토글, 편집/삭제, "새로 추가".
- 편집 폼: 해당 테이블 필드 + accent 선택(색상 팔레트 버튼). 저장 후 공개 페이지에 반영.

### 강사진 관리 `/admin/content/experts` (P2, 명시 요구사항)
- **등록/삭제/수정** 전부 지원. 리스트에서 카드형(사진·이름·역할) 표시, "강사 추가" 버튼.
- 편집 폼 필드: 이름, 역할, 한줄 소개(quote), 태그(multi), accent, **강사 사진 업로드**.
- **이미지 업로드**: Supabase Storage `experts` 버킷에 저장, `image_path`만 DB에 기록.
  - 업로드는 Server Action 경유(admin만). 미리보기 제공, 정사각형 권장·용량/포맷 검증(jpg/png/webp).
  - 공개 페이지 렌더 시 Storage public URL(또는 서명 URL)로 표시, 없으면 도형/이니셜 폴백.
- **삭제 시** DB 행 + Storage 이미지 함께 정리.
- 공개 [experts.tsx](../src/components/sections/experts.tsx) 를 이미지 우선 + 폴백 구조로 수정.

## 7. 공개 사이트 변경점

1. **Final CTA 폼** ([final-cta.tsx](../src/components/sections/final-cta.tsx)): `mailto` → Supabase INSERT.
   - Server Action으로 저장, 성공/실패 토스트, 완료 상태 UI.
   - **저장 성공 시 관리자에게 이메일 알림 발송(P1, 결정 §9).** 문의 유형·세부 유형·발신자·연락처·내용 요약 + 어드민 상세 링크. 발송 실패해도 저장은 성공 처리(알림은 부가).
   - 검증: 필수값, 간단한 스팸 방지(honeypot 등).
2. **콘텐츠 소스 전환(P2)**: 각 섹션이 `site.ts` 대신 서버에서 DB fetch. `site.ts`는 폴백/시드로 유지 → DB 시드 스크립트로 이관.
3. **문의 폼 옵션 소스 전환(P2)**: 유형/세부 유형이 `site.ts`의 `inquiryOptions`(시드) → DB(`inquiry_types`/`inquiry_subtypes`)에서 fetch. 폼은 이미 단일 소스만 읽도록 분리돼 있어 데이터 출처만 교체.

## 8. 기술 메모

- **Supabase 연동**: `@supabase/ssr`로 서버/클라이언트 클라이언트 분리. env: `NEXT_PUBLIC_SUPABASE_URL`, `NEXT_PUBLIC_SUPABASE_ANON_KEY`, (서버 전용)`SUPABASE_SERVICE_ROLE_KEY`.
- **MCP**: CLAUDE.md대로 Supabase 프로젝트 생성 후 `supabase` MCP 등록. API는 `context7`로 최신 문서 확인(@supabase/ssr, Next.js 16 미들웨어).
- **디자인**: 어드민도 [디자인.md](디자인.md) 토큰 사용하되, 공개 페이지의 "포스터" 무드가 아니라 **밀도 높은 실무 대시보드** 톤. Warm Ivory 배경 + flat, 그림자/그라데이션 금지 유지. Shadcn `select`은 이미 설치됨([ui/select.tsx](../src/components/ui/select.tsx)) · table/dialog/form은 추가 필요.
- **가드**: `/admin` 레이아웃 = Server Component에서 세션·admin 확인 후 미인증 시 `/admin/login` redirect. 미들웨어로 세션 리프레시.
- **Storage**: 이미지는 Supabase Storage 버킷 — `experts`(강사) / `work`(포트폴리오 커버) / `insights`(인사이트 커버). 읽기 public(또는 서명 URL), 쓰기/삭제는 admin RLS. 업로드는 Server Action에서 처리하고 DB엔 경로만 저장. **업로드 UI는 공통 컴포넌트로 3곳 재사용**(미리보기·포맷/용량 검증).
- **이메일 알림**: 새 문의 알림 발송용 메일 서비스 필요(예: Resend). env `RESEND_API_KEY`(서버 전용), 수신 관리자 주소는 설정값/`admin_users` 기준. Server Action에서 저장 직후 발송.
- **마이그레이션**: SQL 마이그레이션 파일로 관리(스키마 + RLS + 함수 + 트리거 + 시드 + Storage 버킷/정책).

## 9. 결정 사항 / 남은 이슈

**결정됨 (2026-07-07):**

1. **관리자 = 여러 명.** `admin_users` 테이블 + `is_admin()` 함수로 권한 관리. 화이트리스트 단일계정 방식은 폐기. 첫 관리자는 수동 등록, 이후 어드민에서 관리자 초대/추가 가능(§4.3, P3에서 관리 UI).
2. **콘텐츠는 어드민에서 관리(풀 CMS).** work·insights·testimonials·experts·stats **전부 DB화**. programs·businesses·process·software_categories 도 DB화하되 변경 빈도 낮으므로 **P2 후반**에 이관(그전까지 site.ts 유지). → §4.2 확정.

**결정됨 (2026-07-08):**

3. **문의 폼 세부 유형 = 어드민에서 독립 관리.** 섹션 콘텐츠(programs·businesses)에서 파생하지 않고 전용 테이블(`inquiry_types`/`inquiry_subtypes`)로 CRUD. 프로젝트 문의 세부 유형까지 3유형 일관 관리. "언제나 하드코딩할 순 없다"는 요구에 따라, 코드상 이미 [site.ts](../src/lib/site.ts)의 `inquiryOptions` 단일 소스로 정리(시드) — P2에서 DB로 이관. → §4.2, §5, §3 라우트 반영.

**코드 동기화 (2026-07-08, 기획 이후 진행분 반영):**

4. 문의 폼이 `type` + **`subtype`** 둘 다 제출 → `inquiries`에 `subtype` 컬럼 추가(§4.1).
5. 세부 유형이 label + **`placeholder`**(문의칸 예시) 구조 → `inquiry_subtypes`에 `placeholder` 추가, 세부 6개 초과 시 폼 드롭다운 전환 임계값 존재(§4.2).
6. 강사 3명 실데이터 + 이미지 이미 구현됐으나 **Storage 아닌 `public/experts/` 정적 파일**(필드 `image`) → P2에서 Storage `image_path`로 이관(§4.2). 폴백 렌더는 구현 완료.
7. `select` 등 일부 Shadcn 프리미티브 설치 완료 → §8 갱신.

**결정됨 (2026-07-08, 2차 — 역질문 답변):**

8. **인증은 관리자 전용.** 방문자용 일반 회원 개념 없음, 공개 회원가입 화면 안 만듦. RLS `is_admin()` 단일 게이트로 단순화. → §4.3, §5.
9. **새 문의 시 관리자 이메일 알림.** 문의 저장(P1) 플로우에 포함. 메일 서비스(Resend 등) 연동. → §7, §8.
10. **Work + Insights 커버 이미지.** 두 테이블에 `image_path` + Storage 버킷. 업로드 UI는 강사진과 공통 컴포넌트로 재사용. → §4.2, §8.
11. **Insights 블로그 상세까지.** `slug`/`body`/커버 이미지 추가, 공개 `/insights/[slug]` 라우트, 어드민에 본문 에디터. 작업량 있어 **P2 후반~P3**. → §3, §4.2.

**남은 이슈:**

- 문의 저장 경로: 클라이언트 직접 INSERT(RLS로 제한) vs Server Action 경유 → **Server Action 확정**(검증·스팸 방지·이메일 알림 트리거 위치).
- 인사이트 본문 에디터 형식: 마크다운 textarea(단순) vs 리치 에디터(작업량↑) — P2 착수 시 결정.
- 이메일 수신 관리자 대상: `admin_users` 전체 vs 지정 1~2인 — 발송 연동 시 결정.

## 11. 구축 실행 계획 (병렬)

### 11.1 원칙
- **계약 우선(contract-first)**: §4 데이터 모델을 TS 타입으로 먼저 고정 → 프론트가 실 DB 없이 목업 타입으로 병렬 구축. 통합 지점에서만 실 Supabase에 연결.
- **파일 소유권 분리 + git worktree 격리**로 동시 편집 충돌 방지.
- **Wave0가 유일한 병목** → 최우선으로 짧게 끝내고 병렬 확장.

### 11.2 팀(서브에이전트) 배치
| 에이전트 | 담당 |
|---|---|
| `supabase-backend` | 스키마·마이그레이션·RLS·Auth·Storage·시드·이메일 발송 |
| `nextjs-frontend` | `/admin` 셸·화면, 공개 폼 리팩터, CRUD UI |

### 11.3 Wave 0 — 부트스트랩 (선행, 공유 기반)
병렬 시작 전 반드시 완료. 여기서 모든 스트림의 계약이 확정된다.
1. Supabase 프로젝트 생성 · `.env.local` · `supabase` MCP 등록

### 11.4 Wave 1 — 병렬 (P1 핵심)
| 스트림 | 에이전트 | 산출물 | 파일 소유 |

### 11.7 P2(콘텐츠 CMS) — 동일 패턴 반복

- **Frontend 스트림**: 콘텐츠 CRUD 화면(공통 리스트/폼/이미지 업로드 컴포넌트) — 타입 기반 병렬.

### 11.8 DB 제외 선행 구축 모드 (현재 채택)

**결정**: Supabase 프로젝트 생성·데이터·DB 연결은 **나중에 한 번에** 진행. 그전까지 DB 없이 어드민을 완성 가능한 데까지 만든다.

- **데이터 접근 seam**: 모든 화면은 Supabase를 직접 부르지 않고 **어댑터 인터페이스**(예: `src/lib/admin/data.ts` — `listInquiries()`, `getInquiry()`, `updateStatus()`, 콘텐츠 CRUD 등)만 호출. 지금은 `site.ts` 기반 **mock 구현**을 주입, 나중에 **Supabase 구현으로 교체**(인터페이스 불변).
- **인증 seam**: `/admin` 가드는 `getSession()`/`isAdmin()` 어댑터 뒤에 둠. 개발 중엔 dev-bypass(예: env 플래그)로 화면 접근·검증. 실제 세션은 DB 연결 시 활성화.
- **쓰기 작업(문의 저장·상태 변경·CRUD·이미지 업로드)**: Server Action 시그니처는 확정하되 내부는 no-op/mock(콘솔·인메모리)로 두고, 성공 UX(토스트·상태 전이)까지 완성. 이메일 알림도 인터페이스만, 발송은 stub.
- **지금 만들 수 있는 것**: `/admin` 레이아웃·사이드바, 로그인 UI, 문의 목록/상세/상태 UI, 대시보드, 콘텐츠 CRUD 화면·이미지 업로드 UI(미리보기까지), 공개 폼의 Server Action 배선, shadcn table/dialog/form 도입.
- **나중에 한 번에(통합 체크리스트)**: Supabase 프로젝트·env·MCP → 마이그레이션·RLS·시드 → `@supabase/ssr` 클라이언트·미들웨어 → 어댑터 mock→Supabase 교체 → dev-bypass 제거 → Storage 실업로드 → Resend 실발송 → E2E.

### 11.9 리스크 관리
- **Wave0 병목**: env·타입·의존성이 안 서면 전 스트림 대기 → 최우선.
- **동시 편집 충돌**: worktree 격리 + 파일 소유권 표로 사전 차단, 통합 시 순차 머지.
- **계약 변경**: §4 타입이 도중 바뀌면 전 스트림 영향 → Wave1 중 스키마 변경은 백엔드가 타입부터 갱신 후 공지.
