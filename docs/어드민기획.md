# KPOPSOFT 관리자 플랫폼 설계

갱신: 2026-07-14

## 1. 목적

공개 사이트의 프로젝트 의뢰, 일반 문의, 교육 요청을 안전하게 접수하고 팀원이 함께 처리하는 동시에 랜딩페이지 콘텐츠를 관리하는 한 개의 Next.js 앱입니다. 별도 서버나 NAS, Supabase에 의존하지 않고 Vercel에 앱을 배포하며 Vercel Marketplace Neon과 Vercel Blob을 사용합니다.

## 2. 아키텍처

```text
공개 문의 폼
  -> Next.js Server Action
  -> Zod/honeypot/작성 시간/submissionKey 검증
  -> Neon inquiries 멱등 저장
  -> Cloudflare Email Service -> kpopsoft@gmail.com
  -> Linear -> KPOPSOFT 팀 이슈

Google OAuth
  -> Auth.js
  -> Neon admin_users 활성 계정 확인
  -> /admin 관리 화면

관리자 미디어 업로드
  -> 인증된 token 발급
  -> Vercel Blob
  -> 서명된 callback + Blob 메타데이터 검증
  -> Neon media_assets
```

## 3. 권한 정책

- 방문자용 회원가입은 없습니다.
- Google이 검증한 이메일과 활성 `admin_users.email`이 일치해야 로그인됩니다.
- 활성 관리자는 모두 동일한 최고관리자입니다. 별도 일반 관리자 권한은 두지 않습니다.
- 관리자는 팀원을 추가하고 활성/비활성 상태를 변경할 수 있습니다.
- 마지막 활성 관리자 비활성화는 DB 트랜잭션 잠금으로 거부합니다.
- 권한 변경을 포함한 관리 작업은 `audit_logs`에 남깁니다.
- `ADMIN_DEV_BYPASS=true`는 비운영 환경에서만 작동하며 Production에서는 무시됩니다.

## 4. 화면 범위

| 라우트 | 기능 |
|---|---|
| `/admin/login` | Google 관리자 로그인 |
| `/admin` | 문의 현황 대시보드 |
| `/admin/inquiries` | 검색, 유형·상태 필터, 목록 |
| `/admin/inquiries/[id]` | 원문, 상태, 메모, 메일·Linear 배송 상태와 재시도 |
| `/admin/content/work` | 사례 CRUD, 노출, 정렬, 커버 이미지 |
| `/admin/content/insights` | 인사이트 CRUD, 노출, 정렬, 커버 이미지 |
| `/admin/content/testimonials` | 후기 CRUD, 노출, 정렬 |
| `/admin/content/experts` | 강사 CRUD, 노출, 정렬, 사진 |
| `/admin/content/stats` | 회사 지표 CRUD, 노출, 정렬 |
| `/admin/content/inquiry-options` | 문의 유형·세부 유형·예시 문구 관리 |

모든 화면은 모바일 사이드바/테이블 변형을 포함한 반응형으로 구성합니다.

## 5. 데이터 모델

- `admin_users`: Google OAuth 허용 관리자, 활성 상태, 이름, 아바타
- `audit_logs`: 작업자, 액션, 대상, 메타데이터
- `inquiries`: 유형, 세부 유형, 발신자, 연락처, 내용, 상태, 메모, `submission_key`, 메일·Linear 배송 상태/오류/외부 ID
- `inquiry_types`, `inquiry_subtypes`: 공개 문의 폼의 옵션과 입력 예시
- `work_items`, `insights`, `testimonials`, `experts`, `stats`: 공개 콘텐츠 CMS
- `media_assets`: Vercel Blob URL, pathname, 크기, MIME, 용량, 업로드 관리자

`supabase/migrations/`는 기존 스키마 이력 참고용으로 보존하지만 현재 runtime에서 사용하지 않습니다. 현재 스키마의 단일 소스는 `src/lib/db/schema.ts`입니다.

## 6. 문의 처리 계약

1. 클라이언트가 폼 열림 시간과 UUID `submissionKey`를 만듭니다.
2. Server Action이 필수값/길이, honeypot, 최소 작성 시간, UUID를 검증합니다.
3. Neon이 `submission_key` UNIQUE로 재전송·더블클릭을 하나의 문의로 합칩합니다.
4. 신규 생성일 때만 메일과 Linear 배송을 동시에 시도합니다.
5. 배송 실패는 짧고 정규화된 오류 코드로만 저장하고 문의 본문/토큰/공급자 응답을 로그에 노출하지 않습니다.
6. 관리자는 클라우드 채널을 개별 재시도할 수 있습니다.

## 7. 이메일·Linear

- 기본 메일 수신자는 `kpopsoft@gmail.com`입니다.
- 발신자 주입을 방지하기 위해 고정된 검증 발신자를 사용하고, 유효한 이메일 연락처만 `Reply-To`로 설정합니다.
- 텍스트와 HTML 본문을 함께 만들고 HTML 필드를 escape합니다.
- Linear 이슈에는 문의 ID, 유형, 발신자, 연락처, 본문, 관리자 링크를 담습니다.
- 이미 Linear 이슈 ID가 있으면 중복 이슈를 만들지 않습니다.

## 9. 배포 승인 기준

- 로컬: `npm test`, `npm run lint`, `npx tsc --noEmit`, `npm run build`, `git diff --check`
- DB: 마이그레이션·시드 완료, 관리자 인증/CRUD/마지막 관리자 보호 확인
- Preview: 공개 문의 1회 저장, 중복 제출 멱등성, 메일, Linear, Blob, 모바일 화면 확인
- Production: Google callback/URL, 검증 발신자, 정식 도메인, 관리자 팀 스모크 테스트
- 위 외부 검증이 끝나지 않으면 코드 검증이 통과해도 배포 상태는 **HOLD**입니다.

## 10. 후속 범위

- 교육 신청 템플릿 개편은 현재 데이터 구조를 유지한 채 후속 업데이트로 진행합니다.
- 분산 rate limit/challenge, 백업·복구 운영 절차, 모니터링/알림은 실제 트래픽 전 보강 항목입니다.
