# 개발 상태

마지막 확인: 2026-07-16

## 현재 구현

- Next.js 16 App Router, React 19, TypeScript, Tailwind CSS v4 기반의 KPOPSOFT 반응형 랜딩페이지입니다.
- 첨부 시안에서 확정한 KPOPSOFT 로고와 9개의 고화질 사진, 기존 강사 사진을 유지하며 모바일·태블릿·데스크톱 레이아웃을 구성했습니다.
- 교육은 AI 기초교육, 업무용 AI, 바이브 코딩, AI 자동화, 콘텐츠·동영상 제작, 기업 맞춤·프로젝트 6개 트랙으로 분류했습니다. 교육 신청 템플릿의 추가 개편은 사용자 요청에 따라 후속 업데이트로 보류합니다.

## 운영 플랫폼

- **관리자 인증**: Auth.js + Google OAuth. Google이 검증한 이메일이며 Neon `admin_users`에 활성 관리자로 등록된 계정만 `/admin`에 접근합니다.
- **관리자 권한**: 일반 관리자 기능은 활성 `admin_users`가 사용합니다. 결제 기능은 별도 `billing_admin_roles`의 `BILLING_VIEW | BILLING_EDIT | BILLING_APPROVE | BILLING_REFUND | BILLING_ADMIN` 역할을 추가로 요구합니다. 기존 활성 관리자에게 결제 권한을 자동 부여하지 않습니다.
- **데이터**: Neon Postgres + Drizzle ORM. 문의, 문의 옵션, CMS, 관리자, 감사 로그, 미디어 메타데이터를 저장합니다.
- **미디어**: Vercel Blob 클라이언트 업로드. 관리자 인증, 파일 형식·용량·경로 제한, 서명된 완료 콜백, `head()` 메타데이터 검증을 적용했습니다.
- **문의 접수**: Server Action이 Zod 검증, honeypot, 최소 작성 시간, UUID `submissionKey`를 확인한 후 Neon에 멱등 저장합니다. 중복 제출은 하나의 문의로 처리합니다.
- **알림**: 저장 후 Cloudflare Email Service와 Linear 이슈 생성을 독립적으로 시도합니다. 한 채널의 실패가 문의 저장을 되돌리지 않으며, 관리자 상세 화면에서 상태 확인·재시도·Linear 링크 이동을 할 수 있습니다.
- **개발 우회**: `ADMIN_DEV_BYPASS=true`는 `NODE_ENV !== "production"`에서만 mock 데이터와 로컬 관리자를 활성화합니다. 운영은 설정 누락 시 fail-closed합니다.

## Billing Hub 계약·청구 기반

- **스키마**: `database/migrations/0002_billing_foundation.sql`이 고객사, 담당자, 사이트, 상품, 계약·항목, 생성 실행, 청구서·항목·전송, 결제 관리자 역할의 11개 테이블을 추가합니다. 기존 `0001`은 수정하지 않았습니다.
- **입력·금액 규칙**: 고객사·사이트 코드를 정규화하고 사업자번호 형식과 정확한 HTTPS origin을 검사합니다. 계약과 청구 항목의 공급가액·부가세·합계는 서버에서 다시 계산합니다.
- **계약 상태**: 신규 계약은 `DRAFT`로만 만들고 `ACTIVE | SUSPENDED | ENDED | CANCELED` 변경은 전용 상태 명령과 감사 로그를 거칩니다.
- **자동 초안**: 활성 계약의 다음 청구일을 Asia/Seoul 달력 기준으로 계산하고 `generation_key` unique 및 행 잠금으로 같은 계약·기간의 중복 초안을 방지합니다. 청구서·항목 스냅샷·다음 청구일 이동은 하나의 SQL 경계입니다.
- **승인·이메일**: `DRAFT` 승인 시 금액 재계산, `OPEN` 전환, 승인자·승인시각, 감사 로그, 수신 동의 담당자의 이메일 대기열을 원자적으로 저장합니다. Cloudflare 이메일은 커밋 후 전송하며 실패해도 청구 상태를 되돌리지 않습니다.
- **고위험 재인증**: 승인·무효화는 Google 로그인 시각 기준 15분 이내의 `BILLING_APPROVE` 또는 이를 포함하는 `BILLING_ADMIN` 권한이 필요합니다.
- **기본 상품 시드**: `MAINTENANCE`, `WEBSITE_BUILD`, `ADDITIONAL_DEVELOPMENT`, `EDUCATION`을 삭제 없이 upsert합니다.

### Billing 런타임 설정

```text
BILLING_ENABLED=true
BILLING_CRON_SECRET=<충분히 긴 무작위 서버 시크릿>
BILLING_ADMIN_SEED_EMAILS=<결제 최고관리자 이메일, 쉼표 구분>
BILLING_NOTIFICATION_FROM=<Cloudflare 검증 발신자, 선택>
```

`BILLING_ADMIN_SEED_EMAILS`가 없으면 결제 역할을 누구에게도 자동 부여하지 않습니다. `BILLING_NOTIFICATION_FROM`이 없으면 기존 `INQUIRY_NOTIFICATION_FROM`을 사용합니다. cron 인증은 `Authorization: Bearer ...` 헤더만 허용하며 query parameter로 받지 않습니다.

### 자동 생성 일정

- Route: `GET /api/internal/billing/generate`
- Vercel UTC cron: `15 15 * * *`
- 실행 시각: 매일 00:15 Asia/Seoul
- 응답: 실행 ID와 대상·생성·실패 건수만 반환하며 계약 ID, DB 오류, 시크릿은 반환하지 않습니다.

## Billing 결제 구현

- **무통장 입금**: 서버 설정 계좌만 고객 결제화면에 표시합니다. 최근 `BILLING_APPROVE` 재인증을 통과한 관리자가 청구금액과 동일한 입금액·입금일·입금자·확인근거를 최종 검토해야만 결제, 입금증, `PAID` 청구 상태, 이벤트, 감사 로그가 함께 기록됩니다.
- **토스 결제**: 결제 세션 범위의 청구서를 서버에서 다시 조회해 10분짜리 주문과 멱등 키를 먼저 저장합니다. 성공 URL의 값만으로 결제를 완료하지 않고 저장된 시도와 대조한 뒤 토스 승인 API를 호출합니다.
- **웹훅**: 전송 ID, payload 해시, paymentKey 해시를 receipt-first로 저장합니다. 일반 결제 웹훅은 서명을 가정하지 않으며 Payment API 재조회 결과의 MID·키·주문·금액·상태가 모두 일치할 때만 적용합니다.
- **환불**: 최근 `BILLING_REFUND` 재인증, 내부 잔액과 토스 잔액 일치, 저장된 멱등 키를 요구합니다. 전체·부분 환불을 지원하며 시간초과는 성공이 아닌 `PROCESSING`으로 유지합니다.
- **운영 큐**: `/admin/billing/payments`에서 장기 `CONFIRMING` 결제, `RETRY | REJECTED` 웹훅, `PROCESSING | FAILED` 환불, 완료 결제를 확인합니다. 전체 paymentKey와 공급자 원문 오류는 화면에 표시하지 않습니다.
- **안전한 롤백**: `TOSS_PAYMENTS_ENABLED=false`는 신규 버튼과 주문 생성을 막지만, 서버 키·MID가 유지되면 이미 생성된 결제의 승인 확인·웹훅·재조정·환불은 계속 처리합니다.

## Billing 고객사 위젯

- **삽입 방식**: 고객사 관리사이트에 버전 고정 스크립트 `https://pay.kpopsoft.com/widgets/kpopsoft-billing.v1.js`와 `<kpopsoft-billing>` 요소를 삽입합니다. Web Component는 외부 의존성이 없고 Shadow DOM으로 격리됩니다.
- **표시 정보**: 준비 중, 다음 결제일·예정 금액, 결제 가능, 연체, 결제 완료, 청구 없음 상태를 서버 집계로 표시합니다. 고객사 브라우저가 보낸 금액·결제 상태는 신뢰하지 않습니다.
- **인증**: 고객사 서버가 기존 로그인 세션을 먼저 확인하고 정확한 순서의 HMAC-SHA256 토큰을 발급합니다. 토큰은 120초 이하, exact origin·site·key version·jti replay 제한을 적용합니다.
- **결제 이동**: 위젯은 hash-only 5분 handoff를 발급받아 top-level pay host로 이동합니다. handoff는 원자적으로 한 번만 사용할 수 있고 결제 세션은 30분 idle·2시간 absolute, host-only 암호화 쿠키입니다.
- **호스트 분리**: `www`, `pay`, `admin.pay`의 허용 경로를 Proxy에서 분리하고, `x-forwarded-host`는 Vercel에서만 신뢰합니다. 광범위한 `.kpopsoft.com` 쿠키 Domain은 사용하지 않습니다.
- **연동 관리**: `/admin/billing/integrations`에서 site, 공개 ID, exact origin, key version, 상태, 마지막 사용·회전을 확인합니다. 비밀키는 생성·회전 직후 한 번만 표시하며 회전 시 기존 키가 즉시 무효화됩니다.
- **운영 관측**: 결제 대시보드가 승인 대기, 연체, 전송 실패, `CONFIRMING` 15분 버킷, 웹훅 재처리, 환불 처리/실패, 마지막 자동 청구 실행을 건수로만 표시합니다.

## Toss Payments 연동 기준

2026-07-16에 Toss Payments 공식 문서의 [V2 결제위젯 연동](https://docs.tosspayments.com/guides/v2/payment-widget/integration), [결제 API](https://docs.tosspayments.com/reference), [웹훅](https://docs.tosspayments.com/guides/v2/webhook)을 다시 확인했습니다. V2 SDK의 서버 금액 설정 후 결제 요청, `POST /v1/payments/confirm`, `GET /v1/payments/{paymentKey}`, `POST /v1/payments/{paymentKey}/cancel`, POST 멱등 키, 일반 결제 웹훅 수신 후 Payment API 재조회 원칙을 구현 기준으로 사용합니다. 일반 결제 웹훅에는 임의의 서명 검증을 추가하지 않고 MID·paymentKey·orderId·금액·상태를 공급자 조회 결과와 대조합니다.

- 웹훅 URL: `POST /api/payments/toss/webhook`
- 수동/예약 복구 URL: `GET /api/internal/billing/reconcile` (`Authorization: Bearer BILLING_CRON_SECRET`)
- 복구 대상: 2분 이상 `CONFIRMING` 결제, `RETRY` 웹훅, 2분 이상 `PROCESSING` 환불. 한 번에 최대 100건입니다.
- 현재 로컬 Vercel CLI에는 로그인 자격증명이 없어 연결된 팀의 Pro/Enterprise 플랜을 검증하지 못했습니다. 따라서 10분 간격 cron은 `vercel.json`에 추가하지 않았으며, 자동 복구와 Production 결제 활성화는 플랜 확인 전까지 `HOLD`입니다.

## 배포 전 HOLD

아래 항목은 코드 완료와 별개의 외부 설정입니다. 실제 Preview/Production 검증 전에는 운영 배포 완료로 판정하지 않습니다.

1. Vercel 프로젝트와 GitHub 저장소 연결, `kpopsoft-02.vercel.app` 도메인 확인
2. Vercel Marketplace Neon 생성, `DATABASE_URL` 설정, `npm run db:migrate`/`npm run db:seed` 적용
3. Vercel Blob store 생성 및 `BLOB_READ_WRITE_TOKEN` 연결
4. Google OAuth 클라이언의 Production callback 등록, `AUTH_SECRET`, `AUTH_GOOGLE_ID`, `AUTH_GOOGLE_SECRET` 설정
5. `ADMIN_SEED_EMAILS`에 팀원 이메일을 확정해 초기 최고관리자 시드 적용
6. Cloudflare DNS의 발송 도메인 온보딩, SPF/DKIM, 발신자·수신자 검증, API 토큰 설정. 무료 유지를 위해 검증된 `kpopsoft@gmail.com` 대상 발송 조건을 실제 계정에서 확인
7. Linear API key/team/project 설정 및 KPO-23 연결 스모크 테스트
8. 문의 남용 방지용 공유 rate limit/challenge 공급자 적용. 현재 honeypot·시간 검사는 기본 필터이지 분산 환경의 완전한 남용 방지가 아닙니다.
9. **Billing Preview DB**에 `0002_billing_foundation.sql`을 적용하고 `npm run db:seed`로 기본 상품과 명시적 결제 관리자를 시드한 뒤 마이그레이션·롤백 가능성을 확인. 현재 로컬 작업에서는 외부 DB를 변경하지 않았으므로 `HOLD`입니다.
10. 마이그레이션된 Preview에서 고객사 등록 → 계약 초안 → 활성화 → 자동 청구 초안 → 승인 → 이메일 실패·재시도 브라우저 흐름 확인. 필요한 Preview 데이터와 결제 역할이 아직 적용되지 않아 `HOLD`입니다.
11. Production cron 활성화는 Vercel 배포 소유권, Node runtime, `BILLING_CRON_SECRET`, 운영 DB 마이그레이션과 모니터링 계획 확인 전까지 `HOLD`입니다.
12. **Billing Preview DB**에 `0003_billing_payments.sql`과 `0004_billing_widget.sql`을 적용하고 Preview 전용 테스트 키·MID·웹훅을 등록하기 전까지 카드/간편결제 활성화는 `HOLD`입니다.
